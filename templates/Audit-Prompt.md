# Full Site Audit Prompt / Полный аудит сайта — Универсальный промпт

← [[Templates]] | → [[18-Testing]] · [[30-Security-Checklist]]

> **Скопируй промпт ниже и отправь любому AI-ассистенту на любой проект.**
> Он пройдёт по коду по 10 ролям и выдаст структурированный отчёт `AUDIT-REPORT.md`.
>
> **Copy the prompt below and send it to any AI assistant on any project.**
> It will review the code from 10 roles and produce a structured `AUDIT-REPORT.md`.

## Промпт / The Prompt

````markdown
# Полный функциональный аудит сайта — [НАЗВАНИЕ ПРОЕКТА]

Проведи глубокий аудит сайта по ВСЕМ ролям ниже. Для каждой роли —
пройди по каждому пункту, проверь реальный код, и поставь вердикт:
OK / PARTIAL / MISSING / BUG.
Результат — создай файл AUDIT-REPORT.md с таблицами и пояснениями.

---

## 1. QA / Тестировщик

Проверь ВСЁ, что может нажать, заполнить или сломать пользователь:

- [ ] Все кнопки — есть ли onClick/href? Что происходит при клике?
- [ ] Все формы — валидация (пустой ввод, спецсимволы, XSS-строки),
      loading-состояние кнопки submit, очистка полей после успеха,
      сохранение полей при ошибке
- [ ] Все CRUD-операции — Create / Read / Update / Delete для каждой сущности.
      Составь таблицу: что есть, чего не хватает
- [ ] Confirm перед удалением — есть ли диалог подтверждения?
- [ ] Toast/уведомления — показываются ли при успехе и ошибке?
- [ ] Edge cases — что если 0 элементов? 1000 элементов? Очень длинный текст?
      Пустая строка из пробелов? Двойной клик на submit?
- [ ] Ошибки сети — что если API/БД недоступна? Таймаут? 500 ошибка?
- [ ] Консоль браузера — есть ли ошибки, warnings, deprecated API?

## 2. UX-дизайнер

Оцени с позиции пользователя, который видит сайт впервые:

- [ ] Пустые состояния — каждая страница без данных: есть ли сообщение + CTA?
- [ ] User flow — путь от входа до ключевого действия. Сколько кликов?
      Есть ли тупики (страницы без выхода)?
- [ ] Понятность — все ли кнопки/иконки понятны без подсказок?
      Есть ли tooltips где нужно?
- [ ] Обратная связь — пользователь понимает, что действие выполнено?
      (анимация, toast, смена состояния)
- [ ] Консистентность — одинаковые действия выглядят одинаково?
      Кнопки, отступы, размеры карточек, шрифты
- [ ] Иерархия информации — главное заметно, второстепенное приглушено?
- [ ] Модалки и диалоги — закрываются по Escape и клику вне?
      Фокус возвращается после закрытия?

## 3. Frontend-разработчик

Техническое качество интерфейса:

- [ ] Адаптив — мобайл (375px), планшет (768px), десктоп (1280px+).
      Ничего не обрезается, не уезжает, скролл не ломается
- [ ] Тема светлая/тёмная — все элементы читаемы в обеих?
      Нет белого текста на белом фоне? Нет hardcoded цветов?
- [ ] Акцентные цвета — если есть выбор, все ли варианты работают корректно?
- [ ] Шрифты — загружаются? Есть fallback? Поддержка кириллицы?
- [ ] Изображения — есть alt? Оптимизированы? Lazy loading?
- [ ] Анимации — плавные? Не мешают? Respect prefers-reduced-motion?
- [ ] Компоненты — нет дублирования? Переиспользуются?
- [ ] Hydration — нет мерцания при загрузке? SSR/CSR согласованы?
- [ ] Состояния загрузки — skeleton/spinner на каждой странице с данными?

## 4. Product Owner / Бизнес-логика

Имеет ли всё это смысл с точки зрения продукта:

- [ ] Ключевые сценарии — основные user stories работают от начала до конца?
- [ ] Бизнес-правила — вычисления, статусы, переходы, формулы — корректны?
- [ ] Приоритеты данных — показываем ли самое важное первым?
- [ ] Дедлайны/даты — есть ли предупреждения о просрочке? Сортировка?
- [ ] Роли и доступ — видит ли пользователь только своё?
      RLS / middleware / проверки на месте?
- [ ] Онбординг — новый пользователь понимает что делать?
- [ ] Метрики — можно ли понять, пользуются ли фичей?
      (аналитика, логи, счётчики)

## 5. Инженер безопасности

Защита данных и устойчивость к атакам:

- [ ] Авторизация — есть ли вход, выход, защита роутов?
- [ ] Забыл пароль — есть ли flow восстановления?
- [ ] Смена пароля — есть ли в настройках?
- [ ] XSS — пользовательский ввод экранируется? dangerouslySetInnerHTML?
- [ ] CSRF — защита форм?
- [ ] SQL injection — параметризованные запросы / ORM?
- [ ] Rate limiting — есть ли на auth endpoints?
- [ ] Чувствительные данные — нет ли токенов/ключей в клиентском коде?
      .env в .gitignore?
- [ ] HTTPS — редирект с HTTP? HSTS заголовки?
- [ ] Content Security Policy — настроен?

## 6. SEO-специалист

Видимость для поисковых систем:

- [ ] Meta-теги — title, description на каждой публичной странице
- [ ] Open Graph — og:title, og:description, og:image для шаринга
- [ ] sitemap.xml — существует? Актуален? Доступен?
- [ ] robots.txt — существует? Не блокирует важные страницы?
- [ ] Семантика HTML — заголовки h1-h6 по иерархии? <main>, <nav>, <article>?
- [ ] Canonical URL — нет дублей страниц?
- [ ] Structured data — JSON-LD / schema.org где уместно?
- [ ] 404 страница — кастомная? С навигацией обратно?
- [ ] Редиректы — нет цепочек? 301 а не 302 для постоянных?

## 7. Маркетолог / Конверсия

Приводит ли сайт к целевому действию:

- [ ] CTA — есть ли на каждой странице? Заметен? Понятен?
- [ ] Воронка — от лендинга до регистрации — сколько шагов? Есть ли потери?
- [ ] Аналитика — Google Analytics / Yandex Metrika / Posthog подключены?
- [ ] События — отслеживаются ли ключевые действия? (регистрация,
      создание, завершение)
- [ ] A/B тесты — есть ли инфраструктура?
- [ ] Social proof — отзывы, счётчики, кейсы?
- [ ] Скорость — LCP, FCP, CLS в допустимых пределах?

## 8. DevOps / Производительность

Как сайт работает под нагрузкой и в проде:

- [ ] Core Web Vitals — LCP < 2.5s, FID < 100ms, CLS < 0.1
- [ ] Bundle size — нет ли гигантских зависимостей? Tree-shaking работает?
- [ ] Кеширование — статика кешируется? Cache-Control заголовки?
- [ ] CDN — статика раздаётся через CDN?
- [ ] Ошибки в проде — есть ли error tracking? (Sentry, LogRocket)
- [ ] CI/CD — есть ли пайплайн? Линтер, тесты, деплой автоматизированы?
- [ ] ENV-переменные — разделены dev/staging/prod?
- [ ] Логирование — достаточно для отладки, но без чувствительных данных?

## 9. Accessibility / Доступность

Могут ли пользоваться люди с ограничениями:

- [ ] Клавиатурная навигация — все интерактивные элементы доступны по Tab?
      Видимый focus ring?
- [ ] Screen reader — aria-label, role, aria-describedby где нужно?
- [ ] Контрастность — текст/фон минимум 4.5:1 (AA)?
- [ ] Размер тапа — кнопки минимум 44×44px на мобайле?
- [ ] alt у изображений — описательные, не пустые (кроме декоративных)?
- [ ] Язык страницы — <html lang="ru">?
- [ ] Заголовки — логическая иерархия, не пропущены уровни?
- [ ] Формы — label связаны с input? Ошибки объявляются screen reader?

## 10. Юрист / Compliance

Соответствие законодательству:

- [ ] Политика конфиденциальности — есть? Ссылка доступна?
- [ ] Согласие на ПДн — чекбокс при регистрации?
- [ ] Cookie-баннер — если используются cookie для аналитики?
- [ ] Условия использования — есть?
- [ ] Возрастные ограничения — если применимо?
- [ ] GDPR / 152-ФЗ — возможность удалить аккаунт и данные?

---

## Формат отчёта

Для каждого раздела создай:
1. Сводную таблицу со статусами (OK / PARTIAL / MISSING / BUG)
2. Детальное описание каждого бага или пробела
3. В конце — приоритизированный список рекомендаций:
   - **P0** — блокирует использование
   - **P1** — важно для юзабилити
   - **P2** — улучшения
   - **P3** — nice to have
````
